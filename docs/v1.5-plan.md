# Federal Proposal Assistant — v1.5 Plan

**Date:** 2026-04-22
**Status:** Active plan. Phase A in progress.
**Companion:** [proposal-workbench-v2-architecture.md](proposal-workbench-v2-architecture.md) — the aspirational v2 target state. v1.5 mines that doc for its best ideas and lands them incrementally in the current framework, without rebuilding on a Tauri/SQLite/React stack.

---

## Why v1.5, not v2

The v2 architecture doc correctly identifies limits of pure prompt-first systems (workflow state, approvals, audit, cost control). The v2 direction — structured state + deterministic orchestration + AI as subsystem — is the right long-term target.

But for a single federal BD operator running 5-20 proposals/year, the v2 scope (Tauri + React + Python + SQLite + 10 other libraries) is a 6-12 month engineering project that:
- discards Claude Code's multi-surface leverage (CLI, desktop, web, IDE, plugin marketplace, memory)
- has no migration path for existing active proposals
- commits to DB-backed state before the pain of not having it is proven
- tries to solve "team collaboration" before the user actually has a team

**The v1.5 bet:** most of v2's quality and visibility benefits can be captured by adding **structured sidecars + a read-only dashboard + evidence-grounded drafting** to the existing framework. If after 60 days the remaining pain justifies a full desktop rewrite, we have the data to do it confidently. Until then, ship wins.

---

## Goals

1. Add structured state (JSON sidecars) **without abandoning markdown as the human-readable layer**
2. Make "unsupported claim" detectable and surfaceable — the single biggest quality lever from the v2 doc
3. Provide a read-only local dashboard so pipeline state is visible without running `/status`
4. Track AI cost per workspace per skill
5. Move graphics generation toward parametric templates for the top 5-6 patterns
6. Preserve Claude Code as the AI orchestration layer — don't rebuild it

## Non-goals (explicitly deferred to v2 or later)

- Desktop shell / Tauri / native application
- SQLite as canonical state
- Custom editor component (TipTap / CodeMirror)
- Provider-agnostic AI gateway
- Embedding-based retrieval
- Team collaboration features
- Browser SaaS deployment
- Full rewrite of any existing skill

## User-visible result of v1.5

- Existing commands (`/new-proposal`, `/status`, `/proposal-writer`, `/red-team-review`, `/export-proposal`, etc.) work **unchanged**
- A local dashboard at `http://localhost:8765` shows every proposal's state at a glance
- Every draft claim is linked to evidence; Gold Team auto-flags unsupported claims
- Token spend per proposal per skill is visible on the dashboard
- 80% of new graphics come from parametric templates; 20% free-form

## Phases

### Phase A — Structured sidecars + AI run ledger
**Effort:** ~8-10 hrs | **Dependencies:** none | **Risk:** low (additive)

Foundation. Every later phase consumes these schemas.

**Deliverables:**
- `reference/schemas/compliance-matrix.schema.json`
- `reference/schemas/proposal-plan.schema.json`
- `reference/schemas/activity-entry.schema.json`
- `reference/schemas/ai-run.schema.json`
- `reference/schemas/README.md`
- Per-proposal JSON sidecars: `working/compliance-matrix.json`, `working/proposal-plan.json`, `working/ai-runs.jsonl`
- Skill updates: `proposal-manager` emits `.json` sidecar; `compliance-check` emits `.json` sidecar; CLAUDE.md activity convention extended to require an `ai-runs.jsonl` append on every AI-producing invocation
- Smoke test validates the schemas and their presence in the framework

**Success:**
- `compliance-matrix.json` present in every proposal after one `/compliance-check` run
- Smoke test passes against schemas
- `cat working/ai-runs.jsonl | wc -l` reflects skill invocation count

### Phase C — Evidence ledger + claim-grounded drafting
**Effort:** ~10-14 hrs | **Dependencies:** Phase A | **Risk:** medium (writer output format change, backward-compatible)

Scheduled **before** Phase B because it's the highest-value quality lever. Ties every proposal claim to an evidence item with an explicit ID. Unsupported claims become detectable and Gold Team scores them automatically.

**Deliverables:**
- `reference/schemas/evidence-ledger.schema.json`
- `my-company/evidence-ledger.json` — seeded from existing `my-company/past-performance.md` and `capabilities.md`
- Updated `proposal-writer/SKILL.md` — require `[^EV-###]` citations; mark unsupported claims with `[^CLAIM-UNSUPPORTED]`
- Updated `red-team-review/SKILL.md` Gold Team — scan for unsupported-claim markers → auto-weaknesses
- New skill `/evidence-check` — diffs draft citations vs. ledger; reports missing citations + unused evidence
- Dashboard panel: per-proposal evidence coverage %

**Success:**
- Every draft claim either cited or explicitly marked unsupported
- Gold Team "unsupported claim" finding count goes to zero before submission
- Dashboard shows evidence coverage per proposal

### Phase B — Local read-only dashboard (Streamlit)
**Effort:** ~10-14 hrs | **Dependencies:** Phase A + some of C | **Risk:** medium (first "app" in framework)

**Stack choice: Streamlit.** Not FastAPI + React. Streamlit gives a local URL in <100 lines of Python, auto-refresh, no frontend build step, no auth concerns. Swappable later if scope grows.

**Deliverables:**
- `dashboard/app.py` — Streamlit entry
- `dashboard/views/portfolio.py` — all proposals + due + status + stage + compliance + spend
- `dashboard/views/proposal.py` — per-proposal detail view
- `dashboard/views/spend.py` — aggregate AI cost across proposals
- `dashboard/requirements.txt`
- `dashboard/README.md`
- Optional: `/dashboard` skill that launches it

**Success:**
- Eyeball every proposal's state in <5 seconds without running `/status`
- Answer "which proposal cost the most tokens" in 1 click

### Phase D — Parametric graphic templates
**Effort:** ~4-6 hrs per template × 6 templates = 24-36 hrs, spread over weeks | **Dependencies:** none | **Risk:** low (additive)

Ship parameterized HTML templates for the 6 most common graphic patterns. AI fills labels; layout comes from deterministic templates.

**Delivery order (most-used first):**
1. Three-tier architecture
2. Capability matrix
3. Swim-lane timeline with gates
4. Lifecycle loop
5. Objectives split
6. Differentiator comparison

Each template: `template.html` + `schema.json` + `example.html` + `README.md` under `my-company/graphic-templates/<name>/`.

**Skill update:** `proposal-graphics` gains a template mode — load template + schema, ask for values, render via substitution. Falls back to current AI-HTML-generation flow if no template matches.

**Success:**
- 80%+ of new graphics use a template
- Graphics render in seconds, not minutes
- Consistent brand palette across all proposals

### Phase E — Small quality wins (opportunistic)
**Effort:** ~4-6 hrs total, mixed into other phases

1. Merge `import-from-capture` scaffold into `new-proposal` (shared scaffold function — eliminates drift bug class)
2. Prompt caching in writer and Gold Team (reduces tokens ~30% on long sessions)
3. Diff-based reruns in compliance-check (only re-score changed sections in Gold Team)

## Sequence and effort

| # | Phase | Effort | Why this order |
|---|---|---|---|
| 1 | **A — Sidecars + AI ledger** | ~8-10 hrs | Foundation; enables B and C |
| 2 | **C — Evidence ledger** | ~10-14 hrs | Highest-value; best ROI per hour |
| 3 | **B — Dashboard** | ~10-14 hrs | Read-only visibility of A + C |
| 4 | **D — Parametric graphics** | ~24-36 hrs (spread) | Independent track; run in background |
| ↕ | **E — Small wins** | ~4-6 hrs | Sprinkle in as convenient |

**Total core investment (A + C + B):** ~28-38 hrs — a long weekend or 2-3 focused evenings. Phase D runs calendar-spread as each graphic pattern comes up on real proposals.

## 60-day review criteria

After 60 days (and ideally 2-3 real proposals completed on v1.5 tooling), ask:

- Is markdown drafting hard to coordinate across reviewers? → might justify DB-backed state (early v2)
- Too many proposals for filesystem browsing? → SQLite catalog
- Dashboard scope outgrowing Streamlit? → FastAPI + React
- Multi-user need emerging? → proper team story

If none of those are painful, v1.5 was sufficient. Revisit v2 later, if ever.

---

## What this plan is not

- It is **not** a commitment to eventually build v2. If v1.5 solves the operational problems, v2 stays a backlog item.
- It is **not** a rejection of the v2 doc. It's a staged path *toward* its best ideas using low-risk increments.
- It is **not** a freeze on the current framework. Codex fixes, skill improvements, and operational tweaks continue alongside v1.5 phase work.
